chunklab — the whole product, explained

A full-stack tool to evaluate and compare text-chunking strategies for RAG. This is the single reference: what it does, how the services connect, exactly where every byte is stored, the complete database schema, and how it's operated (logging, rate limiting, reset).

1 · What it is

You create a project, upload documents, then launch a run — a matrix of chunking strategies (sentence, character, recursive, token, semantic, each with size/overlap parameters). For every combination chunklab parses each file, chunks it, embeds the chunks, counts tokens, estimates cost, stores vectors in pgvector, then scores retrieval accuracy with an LLM-as-judge over auto-generated QA pairs. Results feed charts, a comparison table, project-level analytics, and a RAG chatbot.

2 · Learn it one-by-one

A guided path through the codebase — read these in order to understand the whole system.

  1. Config & clients — app/core/config.py (all settings), core/llm.py (Groq), core/embedding.py (FastEmbed + tokenizer), core/redis.py, core/pricing.py, core/ratelimit.py, core/logging.py. Everything else reads config through get_settings().
  2. Data model — app/db/base.py (one MetaData, two schemas), models_core.py, models_results.py, setup_db.py (idempotent bootstrap). See §6.
  3. Parsing — app/services/parsing/: docling primary, pypdf/text fallback.
  4. Chunking — app/services/chunking/: a registry + 5 strategies + the expander. See §8.
  5. Evaluation — app/services/eval/: qa_generator → retriever → judge → metrics. See §9.
  6. The worker — app/workers/run_pipeline.py ties it together; progress.py publishes live updates. See §3.
  7. API — app/api/routers/ (projects, files, runs, progress, results, analytics, chat, logs) + main.py (middleware, rate limits). See §11–12.
  8. Frontend — frontend/src/app/ pages + lib/api.ts, lib/strategies.ts, lib/logger.ts, hooks/useRunProgress.ts.

3 · How a run works (lifecycle)

1
Upload — the API saves the raw file to disk and enqueues a parse job.
2
Parse — the worker runs docling (fallback: pypdf / plain text) → clean text in core.parsed_documents.
3
Expand — combination specs expand into one core.run_combinations row per (strategy × params) cell.
4
QA set — Groq generates grounded Q/A pairs from each document once per run (shared for a fair comparison).
5
Chunk → embed → store — per combination & file: split, count tokens, embed (bge-small), bulk-insert chunks + 384-d vectors into results.chunks.
6
Retrieve → judge → score — per QA pair: pgvector top-k → Groq judge + computed IR metrics.
7
Aggregate — results.combination_stats (tokens/cost/latency) + results.metrics (macro-averaged scores).
8
Explore — dashboards, run compare, project analytics, and the RAG chatbot read these tables.

Progress streams to the browser over SSE, backed by Redis pub/sub (run:{id}:progress) + a snapshot hash (run:{id}:state), and is denormalized onto the DB rows.

Why a separate worker?

Parsing, embedding and dozens of Groq calls are slow. The API enqueues one run_pipeline job; the arq worker (same image, command arq app.workers.settings.WorkerSettings) does the heavy lifting and streams progress.

Why pgvector?

Chunks, embeddings and results live in one Postgres — transactional, joinable. Vectors use an HNSW cosine index; each retrieval is filtered to a single combination's chunks.

9 · Evaluation

LLM-as-judge (Groq, temp 0)

Per question + retrieved context, scored 0–1:

relevancefaithfulnesscontext_precisioncontext_recall

Computed IR metrics

A retrieved chunk is relevant if it shares the same file and ≥50% word overlap with the gold passage.

P@k   = relevant_in_topk / k
recall= 1 if any relevant else 0
MRR   = 1 / rank(first relevant)
nDCG  = DCG / IDCG          (≤ 1)
F2    = 5PR / (4P + R)

10 · Cost model

Local embeddings are free, so embedding cost is a notional reference rate (configurable) that makes combinations dollar-comparable. Groq judge cost is real, from token usage.
embedding_cost = total_tokens / 1000 * EMBED_COST_PER_1K        # notional
judge_cost     = in_tok/1e6 * GROQ_INPUT_COST_PER_M + out_tok/1e6 * GROQ_OUTPUT_COST_PER_M
total_cost     = embedding_cost + judge_cost

12 · Rate limiting

What it is & why

Rate limiting caps how many requests a single client may make to an endpoint in a time window. It exists to:

  • Protect cost — every chat message and every run fans out to the paid Groq API; a runaway loop or a stuck retry could burn the key's quota and money.
  • Protect stability — uploads, embedding and DB writes are heavy; unbounded concurrency could exhaust CPU/connections.
  • Contain abuse / bugs — a misbehaving script (or a frontend bug retrying in a tight loop) is bounded instead of taking the service down.

How it works under the hood

chunklab uses slowapi (a Starlette/FastAPI wrapper over the limits library). The mechanism is a fixed-window counter:

 request ─► key = client IP (slowapi `get_remote_address`)
         ─► bucket = key + route + window ("30/minute")
         ─► counter[bucket] += 1
              ├─ counter ≤ limit  ─► handler runs normally (200)
              └─ counter > limit  ─► RateLimitExceeded ─► HTTP 429 Too Many Requests
         (the counter resets when the 1-minute window rolls over)

So "30/minute" means: for a given IP, the 31st call to that route within the same minute is rejected with 429; once the window resets the budget is full again. Counts are tracked per client IP and per route, so hammering chat doesn't consume your upload budget.

How we wire it (3 pieces)

  1. A shared limiter is created once in app/core/ratelimit.py: 
    limiter = Limiter(key_func=get_remote_address, default_limits=["600/minute"])
  2. main.py registers it: sets app.state.limiter, installs SlowAPIMiddleware (applies the global default to every route), and adds the RateLimitExceeded → 429 exception handler.
  3. Expensive routes add a stricter per-route cap with a decorator (the handler takes a request: Request param so slowapi can read the client IP): 
    @router.post("/chat/stream")
@limiter.limit("30/minute")
async def chat_stream(request: Request, ...): ...

The limits in place

ScopeLimitWhy
Global default (all routes)600 / minuteheadroom for UI polling + SSE
POST /chat/stream30 / minuteeach call hits Groq
POST /projects/{id}/runs30 / minuteruns are expensive (chunk·embed·judge)
POST /projects/{id}/files120 / minuteallows bulk multi-upload

What a client sees, and how the UI copes

A blocked request returns HTTP 429 with a JSON body like {"error":"Rate limit exceeded: 30 per 1 minute"}. The frontend's typed apiFetch surfaces that as an error toast/message, and the chat composer simply shows the message — no crash. Verified live: 35 rapid chat calls → 30×200 then 5×429.

Tuning & scaling. Limits are plain strings ("N/second|minute|hour|day") — change them in ratelimit.py / the decorators. Storage is in-memory, which is correct for a single backend instance. To run multiple backend replicas, give slowapi a shared store: Limiter(..., storage_uri="redis://redis:6379/1") so the counters are global across instances. You can also swap the key function (e.g. per API-key or per-user instead of per-IP).

13 · Run & reset

cp .env.example .env            # set GROQ_API_KEY
docker compose up --build       # postgres, redis, backend, worker, frontend
# open the UI:
open http://localhost:3000
Reset the database (wipe all projects/files/runs): 
docker compose exec postgres psql -U chunklab -d chunklab \
  -c "DROP SCHEMA IF EXISTS core CASCADE; DROP SCHEMA IF EXISTS results CASCADE;"
docker compose exec backend sh -c 'rm -rf /app/data/uploads/*'
docker compose exec redis redis-cli FLUSHALL
docker compose restart backend worker      # init_db() rebuilds the schemas

The frontend talks to the backend via NEXT_PUBLIC_API_BASE_URL (baked at build time). Run exactly one backend so the UI and your data stay in sync.
