Retrieval & Evaluation

How chunklab forms its questions, what retrieves them against the chunks, and how each chunking combination is scored. This mirrors the product-assistant knowledge base (04_qa_generation_and_evaluation.md, 05_retrieval_and_vector_search.md).

Overview

To compare chunking strategies fairly, chunklab builds one shared QA set from your documents, then makes every combination answer the same questions against the same documents. Any difference in score is therefore attributable to the chunking strategy itself.

The pipeline for each question is: embed → retrieve top-k → score. Scoring is done two independent ways — an LLM-as-judge (semantic) and computed IR metrics (objective, vs a gold passage) — then macro-averaged into one scoreboard row per combination.

Embeddings — what the vectors are

Code: app/core/embedding.py. chunklab embeds with FastEmbed running BAAI/bge-small-en-v1.5 — a local, 384-dimensional sentence-embedding model (CPU/ONNX, no API call, downloaded once to a shared cache). FastEmbed L2-normalizes BGE vectors, so cosine distance is exact.

FunctionPurpose
embed_texts(texts) → list[list[float]]Batch-embed chunks at indexing time
embed_query(text) → list[float]Embed a single question at retrieval time

The dimension is fixed at EMBEDDING_DIM = 384 and must match the vector(384) columns and the HNSW index.

QA generation — how the questions are formed

Code: app/services/eval/qa_generator.py; prompt QA_GENERATOR_PROMPT.

1. Sample passages. _sample_passages(text, n, window=900) cuts up to n evenly-spaced ~900-character windows across the parsed document, so questions cover the whole file rather than only the start.
2. Generate one QA pair per passage. For each passage the LLM (Groq llama-3.3-70b-versatile by default) returns strict JSON {"question": "...", "reference_answer": "..."} — a specific factual question answerable from the passage alone, with a grounded reference answer (no yes/no questions).
3. Keep the gold span. Each result is a GeneratedQA holding the question, reference answer, and the source passage + character offsets. That passage is the ground truth for the computed metrics. Stored as results.qa_pairs.
How many? QA_PAIRS_PER_FILE (default 8) per file, capped by MAX_QA_PAIRS_PER_RUN (default 10). The set is generated once per run and reused by every combination — that is what makes the comparison fair. You can also supply your own QA (auto / mine / both).

The retriever — what matches the question to chunks

Code: app/services/eval/retriever.py, function retrieve(session, combination_id, query_vector, k).

1. Embed the question into a 384-d query vector (embed_query).
2. Run a pgvector cosine nearest-neighbour query scoped to one combination:
SELECT id, file_id, content, embedding <=> :qvec AS distance
FROM results.chunks
WHERE combination_id = :cid
ORDER BY distance      -- smallest cosine distance first
LIMIT :k;              -- k = run.top_k (default 5)

In SQLAlchemy: Chunk.embedding.cosine_distance(query_vector), ordered ascending, limited to k.

3. Convert distance to relevance: relevance = max(0, 1 − distance) (≈ cosine similarity, 0..1).
4. Return the top-k chunks; persist the retrieved ids + scores as results.retrievals.

An HNSW index (vector_cosine_ops, m=16, ef_construction=64) makes the search fast and supports inserts without retraining. Filtering by combination_id is what keeps the comparison fair: each combination is only ever searched against its own chunks.

The same retriever powers three things: evaluation (below), the analyst chatbot (over a run's best combination), and the product assistant (over chunklab's own docs in results.doc_chunks).

Scoring the retrieval

(a) LLM-as-judge — app/services/eval/judge.py

The judge LLM receives the QUESTION, the REFERENCE ANSWER, and the numbered retrieved CONTEXT chunks, and returns strict JSON scoring four dimensions 0.0–1.0:

DimensionMeaning
relevanceAre the retrieved chunks on-topic for the question?
faithfulnessIs the reference answer derivable from the context?
context_precisionWhat fraction of retrieved chunks are useful?
context_recallDoes the context contain everything needed to answer?

Scores are clamped to [0,1]; feedback + token usage are kept. Stored as results.judge_evaluations.

(b) Computed IR metrics — app/services/eval/metrics.py

Objective metrics vs the gold passage (no LLM). A retrieved chunk is relevant when it is from the same file as the gold passage AND its word-set overlaps the gold passage by ≥ RELEVANCE_THRESHOLD (0.5). With one gold passage per question there is a single relevant target.

precision@k = (# relevant retrieved) / k
recall@k = 1.0 if any relevant chunk retrieved, else 0.0 (single gold)
MRR = 1 / (rank of first relevant chunk), else 0
nDCG@k = DCG / IDCG,   DCG = Σ 1/log₂(rank+1) over relevant hits, IDCG = ideal ordering (guarantees nDCG ≤ 1)
F2 = (5·P·R) / (4·P + R)   — weights recall above precision

Aggregation

Per-question results are stored as results.query_metrics (the disaggregated / per-question view), then macro-averaged across all questions into one results.metrics row per combination — the judged means + computed-IR means + avg_retrieval_latency_ms. Token/cost/latency totals land in results.combination_stats. The analytics dashboard, the comparison view, and the analyst chatbot all read from these aggregated tables.

Worked example

Combination sentence·512/20, question “What retry settings does the client use?”, gold passage in client.md:

  1. The question is embedded; pgvector returns the 5 nearest chunks from sentence·512/20.
  2. The gold passage's content lands at retrieved rank 2 (word-overlap 0.7 ≥ 0.5, same file) → relevant.
    precision@5 = 1/5 = 0.20 · recall@5 = 1.0 · MRR = 1/2 = 0.50 · nDCG = (1/log₂3)/(1/log₂2) = 0.63 · F2 = (5·0.2·1)/(4·0.2+1) = 0.56
  3. The judge reads the 5 chunks + reference answer → e.g. relevance 0.8, faithfulness 0.9, context_precision 0.4, context_recall 1.0.
  4. These join the other questions' scores; the macro-average becomes sentence·512/20's row in metrics.
