chunklab — Implementation Reference

chunklab is a full-stack tool for evaluating and comparing text-chunking strategies for RAG. You create a project, upload documents, and define a combinatorial matrix of chunking strategies. Each run parses every file, chunks it once per combination, embeds the chunks into pgvector, counts real tokens, estimates cost, and then scores retrieval accuracy with an LLM-as-judge over auto-generated QA pairs. Output feeds analytics dashboards and a RAG chatbot that answers questions over your experiment results.

FastAPI (async) Next.js 15 App Router Postgres + pgvector Redis + arq worker FastEmbed bge-small (384d) Groq llama-3.3-70b Private repo v0.1 / no Alembic

This document is the flagship, end-to-end implementation reference: architecture, the run pipeline, the database schema, every chunking strategy, the evaluation math, the cost model, the full REST API, Docker topology, the security posture, and the architecture decisions behind it all.

3. Goals & Non-Goals

Goals

  • Head-to-head comparison. Run many strategy × parameter combinations against the same documents and the same QA set, so results are directly comparable.
  • Honest measurement. Use the embedding model's own tokenizer for token counts, and pull real Groq token usage for judge cost.
  • Dollar-comparable runs. A notional embedding rate makes otherwise-free local embeddings show up in cost analytics.
  • Two evaluation lenses. Both an LLM-as-judge (relevance/faithfulness/precision/recall) and classic computed IR metrics (P@k, R@k, MRR, nDCG, F2).
  • Live feedback. Background processing with per-file, per-combination progress streamed over SSE.
  • Explorable output. Analytics (accuracy bars, cost-vs-accuracy tradeoff) and a RAG chatbot scoped to a project, run, or run comparison.

Non-Goals (v0.1)

  • Multi-tenant auth. A static user_id ("anonymous") is used; no login/RBAC.
  • Schema migrations. No Alembic — startup bootstrap creates everything idempotently.
  • Hosted/managed embeddings. Embeddings are local (FastEmbed); there is no paid embedding provider.
  • Production-grade retrieval app. The chatbot answers over experiment results; chunklab is a measurement tool, not a production RAG serving stack.
  • Horizontal autoscaling. A single arq worker process is the default; scaling is out of scope here.
  • Streaming embeddings/judge at request time. All heavy work is queued to the worker.

4. Architecture

chunklab is a monorepo (backend/, frontend/, infra/, docs/) deployed as five Docker services. The API layer is thin: request handlers validate input, write rows, and enqueue work. All expensive operations — parsing, chunking, embedding, evaluation — run in a separate arq worker backed by Redis. The backend and worker share the same image and the same Hugging Face model cache.

Backend layout & conventions

  • Settings only via app.core.config.get_settings() (Pydantic Settings). Never read env elsewhere.
  • LLM only via app.core.llm.get_llm(); prompts only in app/prompts/prompt_texts.py.
  • Embeddings / tokenizer only via app.core.embedding.
  • All DB and LLM calls are async; CPU-bound model work is run in a thread.
  • New chunking strategy = a class with split() + label() calling register(...); it auto-appears in runs.
  • New endpoint = a router under app/api/routers/, included in main.py under /api/v1.

5. Run Lifecycle & Data Flow

The heart of chunklab is app/workers/run_pipeline.py. POST /projects/{id}/runs expands the requested combinations, creates the Run + RunCombination rows, and enqueues run_pipeline(ctx, run_id). The worker then executes the following sequence. 

API   POST /projects/{id}/runs ─ expand combinations ─ create Run + RunCombinations ─ enqueue
  │
  └─▶ WORKER run_pipeline(ctx, run_id)
        ├─ set Run status = running
        ├─ load combinations + files
        ├─ FOR each file: ensure ParsedDocument
        │     └─ parse (docling → pypdf/text fallback, in a thread)
        ├─ generate ONE shared QA set (Groq, evenly-spaced ~900-char passages)
        ├─ precompute question embeddings
        │
        ├─ FOR each combination:
        │     ├─ status = chunking
        │     │     └─ FOR each file: split → assemble → count_tokens
        │     │                       → embed (thread) → bulk-insert Chunk rows (+ vectors)
        │     ├─ status = evaluating
        │     │     └─ FOR each QA pair:
        │     │           retrieve top-k (pgvector cosine) → save Retrieval
        │     │           → judge (Groq) → save JudgeEvaluation → compute QueryMetrics
        │     ├─ aggregate CombinationStats (tokens / cost / latency)
        │     ├─ aggregate Metrics (judged means + macro-averaged IR metrics)
        │     └─ status = completed
        │
        └─ set Run status = completed     (errors → failed)

  progress: every stage publishes to Redis  run:{id}:progress  (pub/sub)
            AND updates the  run:{id}:state  snapshot hash for SSE replay
            AND denormalizes progress onto Run / RunCombination rows

Key sequencing decisions

  • Parse once per file. A ParsedDocument is created if absent and reused across all combinations — parsing never repeats per combination.
  • One shared QA set per run. QA pairs are generated once and reused for every combination, so all combinations are scored against identical questions and gold passages. This is what makes the comparison fair.
  • Question embeddings precomputed. The query vectors are computed once before the per-combination loop, not per retrieval.
  • Bulk vector insert. Each file's chunks are embedded and inserted in bulk with their vectors, not one row at a time.
  • Upload-time parsing. parse_file_task(ctx, file_id) is enqueued at upload, so files are typically already parsed before a run starts; the run-time parse step is the safety net.

Progress events

app/workers/progress.py publishes structured events to the pub/sub channel run:{id}:progress and keeps a snapshot in the hash run:{id}:state. The SSE endpoint replays the snapshot first, then streams live events.

Event typePayloadMeaning
run{status, pct}Overall run progress / status transitions.
combo{comboId, label, status, pct}Per-combination progress.
file{comboId, fileId, stage, status, pct}Per-file stage within a combination.
log{level, message}Human-readable log line.

ⓘ Run statuses

A Run moves through queued → running → completed (or failed / partial / canceled). Each RunCombination moves through queued → chunking → embedding → evaluating → completed (or failed).

7. Chunking Strategies

Strategies live in app/services/chunking/ and follow a registry pattern. Each strategy implements split(text, params) -> list[str] and label(params) -> str, and calls register(...) so it auto-appears in run building. There are five built-ins.

StrategyEngineParamsLabel
sentencellama-index SentenceSplittersize (tokens), overlapsentence·{size}/{overlap}
characterManual fixed-size + overlapsize (chars), overlapcharacter·{size}/{overlap}
recursivelangchain RecursiveCharacterTextSplitterchunk_size, overlaprecursive·{chunk_size}/{overlap}
tokenHF tokenizer windowingsize (tokens), overlaptoken·{size}/{overlap}
semanticCustom: sentence embeddings + cosine-distance percentile breakpointsbreakpoint_percentilesemantic·pct{n}

Expansion & assembly

  • expander.expand(combinations) turns each spec {strategy, params} into labeled, de-duplicated ExpandedCombination cells. params may include sizes: [...] to fan out into multiple cells in one go.
  • assemble(text, pieces) wraps raw string pieces into Chunk(index, content, start, end) objects, computing best-effort character offsets back into the source text.

ⓘ Token honesty

Token counts are computed with the embedding model's own Hugging Face AutoTokenizer (with a tiktoken fallback) via app.core.embedding, so reported tokens reflect what the embedder actually sees — not an approximation. This is what makes per-combination token and cost comparisons meaningful.

Adding a strategy

# app/services/chunking/my_strategy.py
from app.services.chunking.registry import register

class MyStrategy:
    name = "my_strategy"
    def split(self, text: str, params: dict) -> list[str]:
        ...
    def label(self, params: dict) -> str:
        return f"my_strategy·{params['size']}"

register(MyStrategy())   # auto-appears in runs; add a UI entry in frontend/src/lib/strategies.ts

8. Evaluation Methodology

Evaluation lives in app/services/eval/ and combines an LLM-as-judge with computed IR metrics. All prompts are centralized in app/prompts/prompt_texts.py (QA_GENERATOR_PROMPT, JUDGE_PROMPT, CHAT_SYSTEM_PROMPT).

8.1 QA generation — qa_generator.py

generate_qa_pairs(text, n) samples n evenly-spaced passages of roughly 900 characters from the document, then calls Groq's structured extract() with QA_GENERATOR_PROMPT to produce {question, reference_answer} per passage. The gold passage and its character offsets are stored on the qa_pairs row. n comes from QA_PAIRS_PER_FILE (default 8). The QA set is generated once per run and reused by every combination.

8.2 Retrieval — retriever.py

retrieve(session, combination_id, query_vector, k) runs a pgvector cosine search: WHERE combination_id = ..., ORDER BY embedding <=> query, LIMIT k. Each result's relevance score is 1 - cosine_distance.

8.3 LLM-as-judge — judge.py

judge(question, reference_answer, contexts) calls Groq's extract() with JUDGE_PROMPT at temperature 0 and returns four scores in [0, 1] plus free-form feedback. Token usage from response.usage is captured for real cost accounting.

DimensionWhat it measures
relevanceDo the retrieved contexts actually pertain to the question?
faithfulnessIs the reference answer supported by (groundable in) the retrieved contexts?
context_precisionOf the retrieved contexts, how much is on-topic vs. noise?
context_recallDo the contexts cover what's needed to answer fully?

8.4 Computed IR metrics — metrics.py

The ground truth is the gold passage a QA pair was generated from. A retrieved chunk counts as relevant when it is from the same file AND its word-set overlap with the gold passage meets a threshold: 

# relevance test (RELEVANCE_THRESHOLD = 0.5)
gold  = words(gold_passage)        # gold needs >= 5 words, else not relevant
chunk = words(chunk_text)
relevant = (chunk.file_id == gold.file_id) and
           (|gold ∩ chunk| / |gold|) >= 0.5

With a single gold passage per question, the per-query metrics over the top-k flags are:

MetricFormula (single gold passage)
precision@knum_relevant / max(k, 1)
recall@k1.0 if any relevant in top-k else 0.0
MRR1 / rank of the first relevant hit (else 0)
nDCG@kDCG / IDCG with DCG = Σ 1/log₂(i+2) over relevant positions, IDCG = 1
F2(5·P·R) / (4·P + R) — recall-weighted F-measure

Per-query results are combined with macro_average(...) (simple mean across queries) and written to the combination's metrics row alongside the judged means. All values are rounded to 4 decimals.

ⓘ Two lenses, on purpose

The judge captures semantic quality ("is this context good?"); the computed metrics capture positional/overlap accuracy against a known gold passage. Reporting both guards against either lens being misleading on its own.

9. Cost Model

Costs are computed in app/core/pricing.py. Local embeddings have no real API cost, so the embedding rate is a notional, configurable reference that makes chunking combinations dollar-comparable. The Groq judge cost is real, derived from response.usage. 

def embedding_cost(total_tokens) -> float:
    return round(total_tokens / 1000 * EMBED_COST_PER_1K, 6)   # notional

def groq_cost(prompt_tokens, completion_tokens) -> float:
    return round(
        prompt_tokens     / 1_000_000 * GROQ_INPUT_COST_PER_M  +
        completion_tokens / 1_000_000 * GROQ_OUTPUT_COST_PER_M, 6)   # real

total_cost = embedding_cost + judge_cost   # stored on combination_stats

⚡ CPU-only torch + shared model cache

Dockerfile.backend installs CPU-only torch first, then the rest of requirements — this stops docling from pulling in the heavy CUDA torch build. System libs libgl1 and libglib2.0-0 are installed for docling, and PYTHONPATH=/app/backend is set. Because backend and worker share the same image and the hf_cache volume, the embedding model is downloaded once and reused by both — no duplicate downloads, faster cold starts.

12. Security

🔒 Secrets never touch git

The real Groq API key lives only in a gitignored .env. The committed .env.example contains placeholders only. The key was copied into chunklab's local .env from another local project and was never committed. The repository is private.

What is ignored

.gitignore blocks secrets while keeping the example:

# .gitignore
.env
.env.*
!.env.example
*.key
*.pem

⚠ Verification gate — run before any git add

# 1) confirm .env is actually ignored (must print a matching rule)
git check-ignore -v .env

# 2) confirm no env file is tracked (must print NOTHING)
git ls-files | grep .env

If step 1 does not match a rule, or step 2 prints any path, stop — do not commit.

Posture summary

  • All configuration is read through app.core.config.Settings; secrets are never hardcoded or read ad hoc.
  • Real .env is gitignored; .env.example ships placeholders for onboarding.
  • Repository visibility is private.
  • CORS is restricted via CORS_ORIGINS (default http://localhost:3000).

14. Testing & Verification

  • Backend unit tests are pure logic and need no running services: python -m pytest backend/tests -v. They cover the deterministic core — chunking strategies, the expander, label generation, and the metric formulas.
  • Lint with ruff check backend/app backend/tests.
  • Frontend build as a typecheck/compile gate: cd frontend && npm run build.
  • End-to-end smoke: docker compose up --build, then seed and walk a run through the UI; SSE progress and the chat stream confirm the worker + pub/sub path.
  • Health: GET /health backs the backend container healthcheck; postgres/redis have their own.

✓ Determinism where it counts

The judge runs at temperature 0 and the IR metrics are fully deterministic given a fixed QA set, so re-running a combination over identical inputs yields stable, comparable scores.

15. Architecture Decisions

DecisionRationale
arq worker for all heavy workKeeps API handlers fast; parsing/embedding/judging can take minutes. Redis already needed for pub/sub.
Two schemas (core / results)Cleanly separates app state from experiment output; results can be reasoned about (and pruned) independently.
Startup bootstrap, no Alembic (v0.1)Idempotent init_db() keeps early iteration simple; migrations can be introduced later without rework of models.
Local FastEmbed embeddingsZero per-embedding cost and no external dependency for the most frequent operation; makes large matrices affordable.
Notional embedding costFree embeddings would otherwise be invisible in analytics; a notional rate keeps combinations dollar-comparable.
Real tokenizer for token countsHonest counts that match what the embedder sees, so size/cost comparisons are trustworthy.
One shared QA set per runEvery combination is judged on identical questions and gold passages — the only way the comparison is fair.
Judge + computed IR metricsTwo complementary lenses (semantic quality vs. positional/overlap accuracy) reduce blind spots.
HNSW (m=16, ef_construction=64)Fast approximate cosine search at small index-build cost; appropriate for interactive analytics.
SSE for progress, stream for chatNative EventSource + Redis pub/sub is simple and robust; chat uses a plain token stream over fetch.
Shared backend/worker image + hf_cacheSingle build, single model download, consistent runtime across API and worker.
Registry pattern for strategiesAdding a strategy is one self-registering class; it auto-appears in runs with no wiring.
CPU-only torch in the imageAvoids docling dragging in multi-GB CUDA wheels; keeps the image lean for CPU deployment.

16. Roadmap / Future Work

  • Alembic migrations to replace the v0.1 startup bootstrap once the schema stabilizes.
  • Authentication & multi-tenancy — replace the static user_id with real users and per-project access control.
  • Pluggable embedding/LLM providers — let runs pick a hosted embedding model or a different judge model, with real per-provider pricing.
  • More chunking strategies — markdown/heading-aware, layout-aware (docling structure), and parent-document / hierarchical chunking.
  • Richer ground truth — multi-passage gold sets and graded relevance for fuller nDCG, beyond the single-gold simplification.
  • Worker scaling — multiple arq workers and concurrency controls for large matrices.
  • Run artifacts / export — downloadable reports (CSV/JSON) and shareable comparison links.
  • Caching of embeddings across combinations where chunk text is identical, to cut redundant embedding work.
