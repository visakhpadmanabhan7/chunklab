1 · Fast mode (skip the LLM judge)

A per-run toggle that drops the judge so a run finishes in seconds instead of being throttled by the Groq free-tier rate limit.

How it works
  • Run builder has a “Run LLM judge” checkbox → enable_judge on the run config.
  • When off, the pipeline skips every per-question judge call and reports only the computed IR metrics (precision@k, recall@k, MRR, nDCG, F2) — which need no LLM (run_pipeline.py).
  • QA generation + retrieval are unchanged; the LLM-judged columns (relevance/faithfulness) are 0 in this mode.
Example
A 1-file, 1-combo run completed in ~12–19 s with 0 judge calls and cost ≈ $0.00009; the same kind of run with the judge on crawls through dozens of throttled 429s. Fast mode is what makes iterating on chunking feasible on the free tier.

2 · Bring-your-own-key / multi-provider

Use Groq, OpenAI, or Anthropic with your own keys, chosen per area (chat · QA-generation · judge). Keys are session-only — saved in your browser, never stored on the server.

How it works
  • A provider registry: get_llm(provider, model, api_key) with adapters for Groq/OpenAI/Anthropic, all exposing chat / extract / stream_chat (core/llm.py). No arguments → the default server Groq.
  • Named keys live only in the browser (localStorage); the chosen key travels with each request and is used transiently. For runs (async), the key rides the job queue (Redis) for that job only — never written to Postgres, never logged.
  • UI: a Settings → API keys page (add label · provider · secret), a picker in the chat panel, and an LLM selector in the run builder.
  • Provider/key errors surface in the chat stream (e.g. ⚠️ LLM error (openai): 401) instead of hanging.
Example
Add a key my-openai · OpenAI · sk-… in Settings → pick it in chat or a run → that request runs on gpt-4o-mini; everything else keeps using the server Groq default. A bad key returns a clean 401 message.

3 · Per-provider pricing

Real LLM cost is computed at each provider/model’s actual token rate, not always Groq’s.

How it works
  • llm_cost(provider, model, prompt_tokens, completion_tokens) looks up a rate table (per 1M tokens) with a provider fallback (core/pricing.py).
  • The pipeline meters the judge with the run’s actual provider+model; embedding cost stays notional (local model = $0, kept for dollar-comparability).
Example
A judge run on gpt-4o-mini is billed at $0.15 / $0.60 per-M (in/out); on claude-3-5-sonnet at $3 / $15; on Groq 8b-instant at $0.05 / $0.08 — so the cost column is meaningful across providers.

4 · Per-question results + selectable metrics

Drill from the per-combination average down to every individual question, and choose exactly which metrics to show / export.

How it works
  • Each question’s computed + judge metrics are persisted in results.query_metrics (per combination × question).
  • GET /runs/{id}/per-question returns the disaggregated rows.
  • Results tab: an Aggregated ⇄ Per-question toggle, metric chips to show/hide columns (they also drive the bar chart), and CSV in both shapes.
Example — a real 4-strategy fast run on SETUP.md

5 · User-provided QA / ground truth

Bring your own questions + reference answers so runs are scored against trusted, domain-specific ground truth — instead of (or alongside) auto-generated QA.

How it works
  • Project-level QA sets in core.project_qa_pairs with CRUD at /projects/{id}/qa-set; a QA-set manager (add / list / delete) in the run builder.
  • Run option Ground truth = auto / mine / both (qa_source). “Mine” spends no QA-gen tokens.
  • Each user row can include a gold passage for the computed metrics; if omitted, it falls back to the reference answer.
Example
Add “How do you start the whole stack?” → “docker compose up --build” to a project, run with Ground truth = both → the run is scored on your question plus generated ones; verified that the user question flowed in and scored (nDCG 0.68, recall 1.0).

6 · Cancel-that-stops-work · Re-run · Redis limiter

Operational robustness for real use.

How it works
  • Cancel actually stops the worker: the pipeline checks the run’s cancel flag between combinations and aborts (previously cancel only marked status).
  • Re-run: POST /runs/{id}/rerun clones a run’s config into a fresh run + a Re-run button on the run page.
  • Redis-backed rate limiter: slowapi counters live in Redis so limits are consistent across backend instances (falls back to in-memory).
Example
Hit Re-run on a finished run → a new queued run with the same matrix appears instantly; cancel a long run → the worker stops at the next combination boundary instead of grinding on.

7 · UI polish

Targeted fixes to the results dashboard and run builder.

How it works
  • Cost-vs-accuracy scatter: per-combination colored + labeled points, Pareto-optimal ones highlighted (★), a metric selector so it’s “cost vs any metric”, a small legend, and a readable cost axis for tiny (fast-mode) costs.
  • Bar chart follows the metric chips and drops all-zero metrics (so judge-off runs don’t show dead relevance/faithfulness bars).
  • Consistent per-combination color across the scatter and the results table (a swatch next to each row).
  • Run builder: advanced options (judge · LLM provider · ground truth) are now collapsed by default to cut scrolling.
Pick cost vs recall@k in the scatter dropdown; deselect relev/faith chips to focus the bar chart on computed metrics; the table dot color matches each scatter point.
