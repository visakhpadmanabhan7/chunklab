ChunkLab

Blog plan · theme · series · template

Chunking is an untuned hyperparameter.

The base plan for writing about ChunkLab: one overarching theme, a six-post series that ladders up to it, and a reusable template.

"Chunking is an untuned hyperparameter — so I built the benchmark for it, in the open."

Every RAG tutorial picks a chunk size (512? 1000? overlap 10%?) and moves on — nobody measures whether the choice was good. ChunkLab is that missing A/B harness: define a matrix of chunking strategies, run them over your documents, and plot retrieval accuracy (LLM-as-judge + IR metrics) against token cost to pick the point on the Pareto frontier you actually want. Each post sells the idea two ways — the product (for RAG practitioners) and the build (for backend / full-stack readers) — both pointing at the open-source repo.

Conventions

Writing

  • Voice — first-person, honest, show the failures (especially #3). No hype.
  • Length — ~1,200–2,000 words. One idea per post; link siblings, don't cram.
  • Code — real snippets from the repo (cite path:line), not pseudo-code.
  • Every post ends with what it is, the repo link, and an invite to try it.

Publishing

  • Primary — dev.to and/or a personal blog.
  • Cross-post — Medium + Hashnode with canonical_url set to the primary.
  • Amplify — LinkedIn + X threads; a Show HN for #1 or #3 once public.
  • Cadence — one post / 1–2 weeks, in order. Lead with #1 and #3.
